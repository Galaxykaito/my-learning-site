# 《Clean Code》读书笔记

## 书籍信息

- **书名**：Clean Code（整洁代码）
- **作者**：Robert C. Martin（Bob 大叔）
- **核心主题**：如何编写易读、易维护的代码

## 核心原则

### 1. 有意义的命名

变量、函数、类的名字应该能够说明它的用途。

**不好的命名：**
```javascript
const d = new Date()
function getData() { }
```

**好的命名：**
```javascript
const currentDate = new Date()
function fetchUserProfile() { }
```

### 2. 函数应该短小

一个函数应该只做一件事，并且做好这件事。

**原则：**
- 函数应该尽量短小（一般不超过 20 行）
- 一个函数只做一件事
- 函数的抽象层级应该统一

### 3. 注释的使用

好的代码应该是自解释的，减少对注释的依赖。

**何时需要注释：**
- 解释复杂的业务逻辑
- 说明为什么这样做（而不是做了什么）
- 法律信息、版权声明

**何时不需要注释：**
- 冗余的注释
- 误导性的注释
- 注释掉的代码（应该删除）

### 4. 错误处理

错误处理很重要，但不应该让它模糊了代码的逻辑。

```javascript
// 使用 try-catch 集中处理错误
try {
  const data = await fetchData()
  processData(data)
} catch (error) {
  handleError(error)
}
```

### 5. 代码格式

团队应该有统一的代码格式规范：
- 缩进
- 空行
- 行宽
- 变量声明位置

## 重要观点

> "代码的阅读次数远大于编写次数，所以要为阅读者优化，而不是为编写者优化。"

> "童子军规则：让营地比你来时更干净。每次修改代码时，都让它比之前稍微干净一点。"

## 实践建议

1. **持续重构**：不要害怕改代码，只要有测试覆盖
2. **遵循 SOLID 原则**：单一职责、开闭原则等
3. **编写测试**：测试是重构的安全网
4. **代码审查**：团队互相审查代码，学习最佳实践

## 总结

整洁代码不是一蹴而就的，需要不断练习和反思。记住：

- **命名要有意义**
- **函数要短小精悍**
- **减少不必要的注释**
- **代码格式要统一**
- **持续改进代码质量**

这本书值得反复阅读，每次读都会有新的收获！
